## 从crawlee到python的类型系统

以下内容是AI从我和朋友的聊天中总结出来的。

---

Crawlee展现了我认为现代爬虫工具应有的设计水平，特别是将爬虫与路由系统结合的架构思路。这种设计让我意识到，好的工具应该从框架层面思考问题，而不是停留在简单的功能堆砌上。

这就是我为什么喜欢TypeScript，因为这个社区里总有人在实践先进的理念（另一个我很喜欢的创意是在serverless运行Chromium爬虫，尽管这些项目无一例外看起来都失败了，但是我还是觉得很有意思）。虽然从语法角度看，TypeScript确实是JavaScript的超集，主要增加了类型系统，但仅仅关注语法层面会错失重点，看一个语言并不能只看语法，ts和js的语法差异是可以忽略不计的，因为TS的额外工作基本都在编译期完成，运行时它们本质上是相同的（尽管JavaScript确实存在多个不同的运行时环境，如V8、Node.js、Bun等）。


聊一个语言不能只聊语法，还要聊生态、性能、发展历史、未来前景、应用领域，语言特性都算小细节了。python告诉我们性能也是无所谓的东西，大多数人根本不关心性能，生态和领域才是最重要的，发展历史和未来前景也是和生态领域强相关的。

对比我之前使用Python的经历，更能体会到TypeScript类型系统的优势。Python的typing模块设计确实存在明显问题，连我这种普通开发者都觉得能设计出更好的方案。更让人失望的是，Python核心贡献者中很多人跑去搞GIL了，typing模块基本没人真正在乎。整个社区对类型系统的态度极其冷淡，你去看typing模块的官方文档就能感受到那种历史包袱和敷衍。

在实际使用中更是痛苦，你会发现用的依赖库几乎没有几个真正支持typing的。开启类型检查吧，一堆第三方库疯狂报错；关闭吧，你自己在代码里写的typing就全报废了。难道你要给每一个用到的库都手动加上类型注解吗？根本不可能！

问题的根源在于Python社区的"pythonic"编程风格——大家习惯了直接传字符串、运行时动态处理那一套，typing只能标一个string，你能注解什么？这种生态习惯导致根本没人在意高级类型功能，也没人愿意推动Python的类型发展。在Python中尝试使用高级类型功能，就像在沼泽里建摩天大楼，不仅地基不稳，周围的人还在嘲笑你为什么要建这么高的楼。

这让我反思评价编程语言应该从哪些维度考虑。除了语法特性，还要看生态系统的成熟度、工具链的完整性、社区活跃度、应用领域的适用性、发展历史、未来前景，甚至包括社区文化。语言特性本身其实只是小细节，生态和领域才是最重要的。就像Python告诉我们的，性能很多时候也没那么重要，大多数人根本不关心，生态和领域才是决定性的。
